# Go 隧道一期开发计划

## 一期目标

先完成“代理 ID 模式”的控制面闭环：

1. 本地 agent 启动
2. 连接公网服务端
3. 服务端分配 `agent_id` 和 `proxy_id`
4. agent 维持心跳在线
5. 服务端可查询在线状态

当前这一步已经有了基础代码骨架。

新增进展：

1. 已加入服务端 TCP 长连接入口
2. 已加入 agent 长连接注册和 `ping/pong`
3. HTTP 在线列表与 TCP 长连接共享同一个内存注册表
4. 已加入服务端标准 SOCKS5 入口
5. 已加入 agent 远程开流、数据转发、关流的基础能力

## 当前文件职责

- `cmd/tunnel-server/main.go`
  - 启动控制面 HTTP 服务
- `cmd/tunnel-agent/main.go`
  - 启动本地 agent，注册并发送心跳
- `internal/server/http.go`
  - 注册、心跳、在线列表接口
- `internal/server/registry.go`
  - agent 在线状态内存表
- `internal/contracts/types.go`
  - 控制面接口结构

## 二期目标

加入真正的隧道转发能力：

1. agent 建立长连接
2. 服务端接收云端代理请求
3. 服务端把请求转给对应 `proxy_id`
4. agent 使用本机网络或上游 S5 出网
5. 数据双向回传

## 二期建议技术路线

### 方案 A：WebSocket 长连接

优点：

- 最容易部署到宝塔 + Nginx
- 走 443，网络兼容性较好
- 便于后期做多端和后台管理

建议：

- agent 到服务端：`wss://域名/ws/agent`
- 每个代理数据流分配 `stream_id`
- 文本帧传控制消息，二进制帧传数据

说明：

当前代码为了先跑通基础链路，使用的是“纯标准库 TCP 长连接”实现。后续如果需要更适合宝塔 + Nginx + 443 的部署形态，再切到 WebSocket。

### 方案 B：HTTP 长轮询

不推荐，只适合作为临时调试手段。

## 三期目标

对接现有 `demo-go-agent` 和 stay：

1. 服务端给每个 `proxy_id` 提供标准代理入口
2. 云端仍然拿到标准格式：
   - `socks5://用户名:密码@服务器公网IP:端口`
3. 现有 `SetProxy` 流程无需大改

## 宝塔部署建议

### 必备

- Ubuntu 22.04 或 Debian 12
- 宝塔
- Nginx
- Supervisor
- 公网 IP

### 端口

- `443`：agent 长连接入口
- `9080`：控制面调试端口，可只内网开放
- `21088-21188`：后续 SOCKS5 入口端口段

## 后续代码拆分建议

下一步新增：

- `internal/server/hub.go`
  - 管理长连接 agent
- `internal/server/session.go`
  - 管理代理数据流
- `internal/server/socks_entry.go`
  - 对外提供标准 SOCKS5 入口
- `internal/agent/tunnel.go`
  - agent 长连接和数据收发
- `internal/agent/dialer.go`
  - 本机直连 / 上游 SOCKS5 出网

## 风险点

1. Windows 环境下构建缓存和临时目录权限可能影响本地编译验证
2. 如果后续引入 WebSocket，建议优先选用稳定的单一库，避免协议层重复造轮子
3. 真正上线前必须补：
   - 鉴权
   - 超时清理
   - 限流
   - 访问日志
   - 强制下线
